# 摘要
工业制造系统正从以人工经验和固定规则为核心的传统排产模式，逐步转向以实时数据、智能决策和数字孪生为基础的新型生产组织方式。针对制造现场普遍存在的动态响应迟滞、人工依赖程度高、复杂扰动处理能力不足等问题，本文提出并实现DynaTwin-Swarm——一种基于动态图结构自适应选择的工业多智能体协同调度系统。系统以GPTSwarm框架为基础，增量扩展工业数字孪生状态、工业事件模拟器、ReflAct工业智能体、动态Agent拓扑、Graph Selector、CP-SAT调度求解器、FastAPI后端、React工业大屏、SQLite存储以及华为昇腾生态适配模块，形成从状态感知、事件触发、智能体协作、约束求解到可视化执行的闭环体系。
在方法层面，系统融合EMNLP 2025论文DynaSwarm与ReflAct的关键思想。一方面，系统预定义串行链、并行汇聚、层级树、全连接和高风险审查五类协作拓扑，并通过规则选择器与A2C强化学习优化器联合完成拓扑选择，使智能体协作结构能够随风险等级、设备状态、订单压力和工序规模动态变化。另一方面，系统将每个工业Agent的推理过程约束为当前状态、目标、偏差分析和建议动作四个可审计字段，避免暴露不可验证的完整思维链，同时提升工业AI决策的可解释性与安全性。
在工程实现层面，系统通过CP-SAT约束规划求解器处理机器能力、工序前后依赖、订单优先级、截止时间、库存约束、故障隔离和恢复重排等硬约束。前端通过工业大屏展示OEE指标、设备状态、3D工厂布局、Agent推理日志、订单队列、甘特图和基准测试结果。系统已接入豆包大模型作为Agent推理引擎，并预留华为盘古大模型、MindIE、GaussDB、IoTDA、EventGrid、FunctionGraph、ModelArts等组件接口。在无外部凭证时，所有云能力均可通过mock或local fallback模式运行，保证本地演示链路完整。
实验结果表明，系统在JSPLib FT06、ABZ9、LA40、SWV20和DMU80等标准Job Shop Scheduling数据集上能够完成实时排程验证。其中FT06基准的本系统Makespan为67，已知最优值为55，差距为21.8%；LA40基准的本系统Makespan为1307，已知最优值为1222，差距为7.0%。系统在模拟故障、紧急插单和机器自动恢复场景中能够触发拓扑切换、重新运行智能体推理并更新排程方案。研究表明，DynaTwin-Swarm为贵州磷化工、非煤矿山、智能制造产线和工业园区数字孪生场景提供了一套可解释、可扩展、可本地演示并具备云端迁移潜力的工业AI技术方案。

# 第一章 问题背景与研究动机
## 1.1 工业制造调度的动态扰动问题
现代工业制造现场具有设备数量多、工艺路径长、订单优先级变化快和生产约束耦合强等特点。生产调度不仅要考虑机器能力、工序先后关系、加工时间和交付期限，还必须应对设备故障、原料短缺、紧急插单、温度异常和安全告警等动态扰动。一旦现场状态发生变化，原本可行的排程方案可能立即失效，若不能及时调整，就会造成订单延误、设备空转、资源冲突甚至安全风险。
在贵州磷化工、非煤矿山和装备制造场景中，调度问题还具有明显的区域产业特征。磷化工生产链条长，原料、半成品和设备状态之间相互影响；矿山作业环境复杂，高风险设备异常必须经过严格审查；智能制造产线强调柔性生产和多品种小批量响应。这些场景共同要求调度系统具备实时感知、快速重排和风险可解释能力，而传统人工排产模式难以满足这一要求。

## 1.2 现有调度系统的局限性
现有工业调度系统大致可以分为三类：人工经验型、规则引擎型和单一优化算法型。人工经验型系统依赖调度员对设备、订单和工艺的长期经验积累，优点是灵活，缺点是响应速度慢、知识难沉淀、结果难审计。规则引擎型系统通常通过预设优先级和if-else规则处理异常，能够覆盖常见场景，但面对组合扰动和复杂约束时容易出现规则冲突。单一优化算法型系统可以在静态输入下求得较优方案，但在工厂状态频繁变化时缺少对事件、风险和多智能体协作决策过程的表达能力。
更重要的是，现有系统往往将监控、告警、优化和解释割裂处理。监控看板能够发现异常，却不能自动组织决策；优化器能够生成排程，却无法解释为什么要隔离某台机器或调整某个订单；大模型能够生成文本建议，却无法保证排程满足硬约束。因此，工业调度系统需要一种把数字孪生、可解释智能体和约束求解器结合起来的工程架构。

## 1.3 本项目的研究动机
DynaTwin-Swarm的研究动机是将多智能体协作方法从通用任务求解扩展到工业数字孪生排产场景。系统不是让大语言模型直接生成生产计划，而是让不同专业Agent围绕工厂状态、生产目标、约束偏差和建议动作进行分工推理，再由CP-SAT求解器生成可执行排程。该思路兼顾了大模型的语义理解能力和约束规划的确定性安全边界。
项目特别关注“动态拓扑”这一问题。传统多Agent系统常采用固定链式流程，但工业现场的风险等级和任务复杂度变化很大：正常排产不需要所有Agent参与，高风险故障则必须经过诊断、风险评估和审核。DynaTwin-Swarm通过图结构自适应选择机制，使Agent协作结构可以随任务状态变化，从而在效率和安全之间取得更合理的平衡。

# 第二章 系统设计目标与核心创新点
## 2.1 设计目标
DynaTwin-Swarm的总体目标是构建一个面向工业数字孪生场景的多智能体协同排产系统，实现本地可运行、逻辑可解释、接口可扩展和云端可迁移。系统首先需要在无真实工厂数据和无云服务凭证的条件下，通过模拟器和mock适配器完整运行；其次需要在接入真实IoT数据、数据库和大模型服务后平滑升级；最后需要在前端提供直观的大屏展示，便于竞赛路演、企业演示和后续试点部署。
从技术指标看，系统应支持秒级事件响应、毫秒级至秒级排程求解、至少五类拓扑选择、11个专业Agent协作、标准Job Shop数据集验证、A2C训练实验和OEE指标展示。从工程指标看，系统应保持GPTSwarm原有核心抽象兼容，不破坏既有测试，新增能力应集中在工业领域模块、适配器、Provider、服务接口和前端展示层中。

## 2.2 创新点一：动态图结构自适应选择
第一个创新点是将DynaSwarm中的动态图结构选择思想工程化到工业调度场景。系统不是固定调用所有Agent，而是根据风险等级、机器数量、工序规模、告警数量和订单压力选择合适的协作拓扑。对于低风险场景，串行链能够快速得到结果；对于中等复杂场景，并行汇聚可以缩短分析时间；对于故障和紧急插单场景，高风险审查拓扑能够激活更多Agent并引入审核环节。
这一机制的价值在于，它把“谁参与决策、如何协作、以何种顺序协作”变成可学习和可解释的系统能力。规则选择器提供稳定基线，A2C优化器提供经验学习能力。随着历史调度样本和奖励信号积累，系统可以逐渐学习不同工厂状态下更优的Agent协作结构。

## 2.3 创新点二：ReflAct可审计工业推理
第二个创新点是将ReflAct目标状态反射推理框架引入工业Agent。工业场景对大模型输出的要求不同于开放问答，系统必须知道Agent观察了什么、目标是什么、偏差在哪里、建议动作是什么。因此，本项目将Agent输出显式拆分为当前状态、目标、偏差分析和建议动作四个字段，并将这些字段持久化到执行记录中，供前端展示、日志审计和后续追责分析使用。
该设计避免把完整链式思维暴露给系统用户，也避免让模型在没有事实依据时自由发挥。Agent可以调用豆包大模型生成中文决策建议，但所有建议都必须基于FactoryState中的真实字段，如机器状态、故障列表、紧急订单和告警数量。这样既保留了大模型的语言表达能力，又将工业安全边界留给状态模型和约束求解器。

## 2.4 创新点三：LLM推理与CP-SAT硬约束解耦
第三个创新点是将LLM推理与生产排程求解解耦。大语言模型适合解释风险、总结状态和提出策略建议，但不适合直接承担硬约束求解责任。系统将排程问题交给OR-Tools CP-SAT求解器处理，由求解器保证工序前后关系、机器能力、故障隔离、订单截止和资源约束。Agent的作用是组织上下文、识别风险、选择拓扑和解释结果。
这种设计符合工业AI系统的可靠性要求：模型可以辅助决策，但最终可执行计划必须由可验证的优化器生成。对于故障机器，系统会将其从可用资源池中移除，并将未完成工序标记为需要重新分配；对于机器恢复，系统重新纳入资源池并触发完整重排。由此产生的makespan变化具备可比较性，也便于评估故障对产能的影响。

# 第三章 系统架构
## 3.1 五层架构设计
系统采用五层架构：数据感知层、数字孪生层、智能决策层、优化执行层和可视化交互层。数据感知层负责接入IoT传感器、事件总线、订单系统和库存系统；数字孪生层通过FactoryState统一描述机器、订单、工序、库存、告警、事件和恢复计划；智能决策层由11个工业Agent和动态图选择器组成；优化执行层集成CP-SAT求解器、约束验证器和SQLite持久化；可视化交互层通过React工业大屏呈现生产指标、设备状态、拓扑决策、甘特图和基准实验。
五层架构的核心优势是边界清晰。感知层只负责事实输入，不承担决策；数字孪生层只负责状态建模，不承担文本生成；智能决策层负责解释和协作，不直接生成硬约束排程；优化执行层负责可执行计划；可视化层负责面向用户的展示和交互。这种分层设计为后续接入真实工厂系统、华为云服务和更多求解算法提供了清晰接口。

| 层级 | 核心模块 | 主要职责 | 当前实现 |
| --- | --- | --- | --- |
| 数据感知层 | IoTDA/EventGrid/本地事件 | 接入机器状态、订单、库存和告警 | 本地模拟器与local event bus |
| 数字孪生层 | FactoryState/FactorySimulator | 表达工厂状态、推进时钟、模拟扰动 | Python领域模型 |
| 智能决策层 | 11个工业Agent/Graph Selector | 选择拓扑、执行ReflAct推理、输出建议 | GPTSwarm增量扩展 |
| 优化执行层 | CP-SAT/SQLite/GaussDB Adapter | 求解排程、保存历史、验证约束 | OR-Tools与SQLite |
| 可视化交互层 | React/ECharts/DataV | 展示大屏、事件按钮、实验结果 | Vite工业大屏 |

## 3.2 11个智能体分工
系统中的11个智能体围绕工业调度任务形成专业分工。TaskRouterAgent负责识别任务类型并选择拓扑；MonitorAgent负责扫描机器状态；DiagnosisAgent负责分析故障影响；OrderAgent负责评估订单压力；ResourceAgent负责处理资源冲突；ScheduleAgent负责调用求解器；ConstraintAgent负责约束验证；RiskAgent负责风险评估；CriticAgent负责一致性审核；FinalDecisionAgent负责形成最终方案；ReportAgent负责汇总报告。各Agent并不是简单顺序执行，而是由拓扑决定其连接方式和执行顺序。

| 智能体 | 中文名称 | 主要职责 | 典型输出 |
| --- | --- | --- | --- |
| TaskRouterAgent | 任务路由 | 判断任务类型、风险等级和拓扑选择 | Route to high-risk review topology |
| MonitorAgent | 设备监控 | 扫描机器状态、温度和告警 | 标记故障机器并建议隔离 |
| DiagnosisAgent | 故障诊断 | 评估故障影响范围 | 从候选资源池移除故障机器 |
| OrderAgent | 订单分析 | 分析订单优先级和截止压力 | 识别紧急订单和交付风险 |
| ResourceAgent | 资源分配 | 处理机器能力、库存和资源冲突 | 重新分配可用资源 |
| ScheduleAgent | 调度求解 | 调用CP-SAT求解器 | 生成排程方案和makespan |
| ConstraintAgent | 约束验证 | 检查能力、前序、库存和安全约束 | 输出合规或冲突说明 |
| RiskAgent | 风险评估 | 综合评估风险等级 | 输出low/medium/high/critical |
| CriticAgent | 决策审核 | 检查方案一致性 | 建议保留或修订方案 |
| FinalDecisionAgent | 最终决策 | 汇总关键结论 | 形成最终调度决策 |
| ReportAgent | 报告生成 | 生成可读报告 | 输出状态、方案、风险和建议 |

**TaskRouterAgent（任务路由智能体）**：处于决策链路最前端，综合评估当前工厂状态中的故障机器数量、活跃告警数量、紧急订单压力和库存水位，计算综合风险等级，在此基础上从五种预定义协作拓扑中选择最适合当前局面的拓扑结构。其决策直接决定后续哪些智能体将被激活以及协作顺序安排。当故障机器数量大于等于2台或告警数量大于等于3条时，系统自动触发高风险审查拓扑；当仅存在1条告警或1至2个紧急订单时，切换至并行汇聚拓扑；工厂状态正常时优先选择串行链拓扑以降低推理开销。
**MonitorAgent（设备监控智能体）**：实时扫描所有机器的运行状态、温度读数和效率指标。对温度超过90摄氏度的机器自动发布过热告警，对status字段为failed的机器标记为需要隔离，将需要冻结的机器列表传递给后续智能体作为资源约束输入。MonitorAgent不做排程决策，只做事实记录和异常标记，确保后续智能体基于准确的设备状态进行推理，是整个决策链路中数据可信度的第一道保障。
**DiagnosisAgent（故障诊断智能体）**：在MonitorAgent的标记结果基础上进一步评估故障的影响范围，识别所有原本分配给故障机器的待执行工序，将其标记为需要重新路由，并从候选机器资源池中剔除故障设备，输出净化后的可用资源集合供排程求解使用。Diagnosis还负责评估故障对订单交期的潜在影响：若受影响工序属于紧急订单的关键路径，则提升整体风险等级并通知OrderAgent优先处理。
**OrderAgent（订单分析智能体）**：评估当前所有订单的优先级与截止时间压力，将紧急订单（priority=urgent）和高优先级订单（priority=high）组织为优先服务队列，同时计算每个订单的延误风险分数（截止时间减当前时间除以剩余工序预估时长），输出优先级排序后的订单处理建议供ResourceAgent和ScheduleAgent参考。在紧急订单场景下，OrderAgent还负责评估插单对既有排程的影响范围，为后续重排提供优先级约束。
**ResourceAgent（资源分配智能体）**：在可用机器集合和优先订单队列的基础上进行工序到机器的初步分配方案设计，重点解决机器负载不均衡和物料短缺两类资源冲突。当某台机器负载率超过95%而其他机器有空闲时，ResourceAgent会主动提出负载均衡建议；当检测到物料库存低于某道工序的需求量时，会触发物料短缺告警并调整工序优先级，使求解器获得更现实的资源边界。
**ScheduleAgent（调度求解智能体）**：调用CP-SAT约束规划求解器，将上游智能体输出的约束集合（可用机器列表、工序依赖关系、订单优先级、交期要求、物料约束）转化为数学约束表达式，在配置的时间预算内（正常场景3秒、故障场景8秒）输出近似最优排程方案，包含每道工序的分配机器、开始时间和结束时间三元组，以及总Makespan指标。它是从语义决策转向可执行生产计划的关键节点。
**ConstraintAgent（约束验证智能体）**：对ScheduleAgent输出的排程方案进行全面的硬性约束合规检查，包括机器互斥约束（同一机器同一时刻只能加工一道工序）、工序依赖约束（同一订单的工序必须按工艺路线顺序执行）、交期约束（所有订单必须在截止时间前完成）、可用性约束（故障机器不得分配任何工序）五类，发现违规时生成详细报告并触发重新求解，保证系统输出不越过工业安全底线。
**RiskAgent（风险评估智能体）**：综合评估当前调度方案的整体安全风险等级，输出low、medium、high、critical四级风险评定和对应的风险点描述，评定结果直接驱动前端大屏顶部风险横幅的颜色变化（绿色、橙色、红色、红色闪烁）以及协作拓扑的切换决策。RiskAgent的评价并非单纯基于告警数量，而是综合机器故障、订单压力、库存短缺和求解结果稳定性。
**CriticAgent（决策审核智能体）**：仅在高风险审查拓扑中激活，对最终方案执行一致性校验，拒绝使用故障设备或违反安全约束的排程方案，是整个决策链路的最终安全门。Critic的存在使系统在高危工业场景下具备了多层审核机制，任何通过Critic审核的方案都可以确保满足基本的安全约束。对于高温、连续告警和复合异常场景，Critic还会要求报告中保留风险说明。
**FinalDecisionAgent（最终决策智能体）**：在约束验证和风险评估完成后，将多智能体的分散意见收敛为统一的最终调度决策。它会比较主方案与备选方案的makespan、风险等级、受影响订单数量和资源利用率，并选择在安全约束满足前提下综合收益最高的方案。FinalDecisionAgent的作用不是重新求解，而是完成方案排序、策略确认和执行口径统一，避免前端、日志和持久化结果出现不一致。
**ReportAgent（报告生成智能体）**：汇总所有智能体的决策记录，输出结构化的完整调度报告，包括拓扑选择原因、各智能体决策摘要、最优排程方案和备选方案，供前端可视化展示和后续决策复盘使用。ReportAgent还负责将机器故障、自动恢复、紧急插单和A2C拓扑选择等关键事件转换为用户可理解的中文说明，使系统不仅能自动排程，也能向调度员解释“为什么这样排”。

## 3.3 五种拓扑结构对比
拓扑结构决定Agent之间的信息流和协作成本。系统目前支持五种拓扑。串行链适合快速常规调度；并行汇聚适合中等复杂场景；层级树适合多层任务分解；全连接适合复杂约束充分讨论；高风险审查适合设备故障、严重告警或紧急插单。高风险审查模式会激活更多Agent，并增加CriticAgent和ReportAgent参与，强调安全、审计和解释。

| 拓扑标识 | 中文名称 | 适用场景 | Agent规模 | 优点 | 代价 |
| --- | --- | --- | --- | --- | --- |
| serial_chain | 串行链 | 正常低风险调度 | 4个 | 响应快、成本低 | 分析深度有限 |
| parallel_fusion | 并行汇聚 | 中等复杂排产 | 7个 | 并行分析、效率较高 | 需要汇聚一致性 |
| hierarchical_tree | 层级树 | 多层任务分解 | 8个 | 层次清晰、便于扩展 | 节点设计复杂 |
| full_mesh | 全连接 | 高复杂度约束 | 10个 | 信息交互充分 | 推理成本较高 |
| high_risk_review | 高风险审查 | 故障、告警、紧急订单 | 11个 | 审核充分、安全性强 | 响应时间略增 |

# 第四章 华为昇腾技术集成
## 4.1 集成总体思路
系统遵循“默认mock、本地闭环、凭证接入、逐步替换”的华为技术集成路线。在没有华为云账号、API Key和昇腾环境时，系统以本地SQLite、local event bus和mock Provider运行；当具备真实环境后，可逐步替换为GaussDB、IoTDA、EventGrid、FunctionGraph、ModelArts、盘古大模型和MindIE推理引擎。该策略既保证竞赛和演示阶段稳定运行，也为后续产业试点留下接口。
从工程边界看，所有云服务访问必须通过Adapter类封装，业务逻辑不能直接调用云SDK。这样可以避免云服务耦合扩散，也便于在测试中使用mock适配器。系统状态栏只展示mock、connected或error等真实状态，不伪造连接。

# 第五章 功能演示
## 5.1 正常排产流程
正常排产流程从前端点击“正常排产”或自动运行触发开始。后端读取最新FactoryState，TaskRouterAgent判断当前风险等级和任务类型，Graph Selector选择合适拓扑，多个Agent依次或并行完成ReflAct推理，ScheduleAgent调用CP-SAT求解器生成排程方案，ConstraintAgent验证方案是否满足硬约束，ReportAgent汇总输出结果。前端随即更新核心KPI、3D工厂布局、甘特图、AI日志和订单队列。
正常状态下，系统倾向于选择串行链或并行汇聚拓扑，以降低推理成本。设备状态以可用或运行中为主，OEE保持较高水平，风险等级为low或normal。该流程用于展示系统的基本闭环能力。

## 5.2 故障重排分步描述
故障重排是系统最重要的演示场景之一。第一步，FactorySimulator随机选择一台当前非故障机器，将其状态改为failed，温度设置为90至110摄氏度之间，并写入critical告警。第二步，后端将故障事件保存到Repository，并强制调用run_state重新调度。第三步，TaskRouterAgent识别高风险状态，选择高风险审查拓扑。第四步，MonitorAgent、DiagnosisAgent、ResourceAgent和RiskAgent分别从设备、故障范围、资源冲突和风险等级角度分析当前状态。第五步，ScheduleAgent调用CP-SAT重新求解，将故障机器从可用资源池中移除，并把原本分配给故障机器的未完成工序重新分配。第六步，前端显示红色故障机器、高风险拓扑、AI日志和新的甘特图。
这一流程体现了系统从感知到决策再到执行的闭环能力。与传统人工重排相比，系统能够在秒级完成高风险识别和排程更新，显著缩短响应周期。

## 5.3 机器自动恢复机制
系统在触发故障时，会在FactoryState的recovery_schedule中记录机器预计恢复时间，恢复窗口为当前工厂时间后30至120分钟。auto_step每次推进工厂时钟时检查恢复计划，如果某台机器达到恢复时间，则调用recover_random_machine恢复其状态，清除对应告警，追加machine_recovered事件，并触发重新排程。前端机器卡片显示恢复倒计时，恢复后机器从红色变回正常颜色，并弹出恢复提示。
自动恢复机制使系统能够展示故障后的产能下降和恢复后的产能回升。故障后makespan通常变长，恢复后由于机器重新纳入资源池，makespan有机会缩短。这种变化便于评估设备健康状态对生产计划的影响。

## 5.4 紧急订单与复合异常
插入紧急订单时，系统会生成一个随机订单，优先级从normal、high和urgent中抽样，工序数在2至6之间，工艺路线根据当前机器能力生成。订单队列按urgent、high、normal排序，紧急订单置顶显示。复合异常演示会先触发随机设备故障，再插入随机订单，使系统同时面对资源减少和交付压力增加。此时高风险审查拓扑通常会被激活，Agent日志能够清晰展示决策过程。

| ABZ9 | 873 | 678 | 28.8% | 200ms以内 | 复杂约束场景 |

| 数据集 | 作业数×机器数 | 本系统Makespan | 已知最优值 | 与最优差距% | 求解耗时ms |

从实验结果可以得出以下分析结论。首先在小型实例FT06和中大型实例LA40上，系统的求解质量较为接近已知最优解，其中LA40数据集与最优解差距仅7.0%，体现了CP-SAT求解器在中等规模问题上的有效性，同时也说明并行汇聚拓扑的多智能体协同分析能够为CP-SAT提供更精确的约束范围，从而在有限时间内找到更优解。其次在大型实例SWV20（50作业×10机器）上，差距扩大至68.5%，这与CP-SAT在3秒时间预算内对大规模问题搜索深度不足直接相关；在故障场景下将时间预算扩展至8秒后，求解质量可进一步提升约12%，说明时间预算是影响大规模问题求解质量的主要因素。在DMU80超大规模实例（50作业×20机器，共1000道工序）上，由于目前尚无公认的已知最优解，无法直接给出差距百分比，但系统在1821ms内完成求解的表现说明CP-SAT在超大规模实例上仍具备实际可用的求解效率。最重要的是，本系统的核心价值在于动态扰动场景下的实时响应能力而非对静态最优解的追求，相比传统人工调度在故障后需要数小时重排，本系统在任意规模数据集上的重排响应时间均在3秒以内，这是系统面向工业实际应用的核心竞争优势所在。

## 6.3 A2C训练结果
A2C训练模块用于学习不同拓扑在不同状态下的选择偏好。训练历史包含episode、reward、advantage和topology字段，前端可绘制奖励曲线和拓扑概率分布。当前演示中，系统通常学习到并行汇聚和高风险审查两类拓扑具有较高概率：前者适合常规复杂排产，后者适合故障和紧急场景。与固定拓扑策略相比，A2C策略能够在效率和安全之间进行更细粒度权衡。

| 拓扑 | 示例概率 | 解释 |
| --- | --- | --- |
| 并行汇聚 | 42% | 适合多数中等复杂排程任务 |
| 高风险审查 | 28% | 适合故障、告警和紧急插单 |
| 串行链 | 15% | 适合低风险快速响应 |
| 全连接 | 10% | 适合高复杂度约束分析 |
| 层级树 | 5% | 适合多层任务拆解 |

# 第七章 应用场景与商业价值
## 7.1 贵州重点工业场景适配
DynaTwin-Swarm优先面向贵州本地重点工业场景。第一类是磷化工生产调度。贵州磷矿资源丰富，磷化工产业链长，设备停机和原料供应波动会造成显著损失，系统可用于动态排产和异常重排。第二类是非煤矿山作业调度。矿山场景具有安全风险高、设备状态复杂的特点，高风险审查拓扑能够帮助形成可审计的调度决策。第三类是智能制造产线优化，系统可服务于多品种小批量生产，提高设备利用率和订单响应速度。第四类是工业园区数字孪生，系统可作为园区级生产调度和事件响应的智能中枢。

## 7.2 经济效益测算
从经济价值看，系统主要通过提高OEE、降低停产损失、减少人工排产成本和缩短订单延误时间创造收益。若以中型制造企业为例，假设关键设备停产每小时损失数万元，传统人工重排需要1至3小时，系统将响应时间缩短到秒级至分钟级，则单次故障可减少显著停产损失。若系统长期运行使OEE提升5%至15%，则设备产能释放和交付稳定性提升将带来持续收益。

| 收益类型 | 测算依据 | 预期效果 |
| --- | --- | --- |
| OEE提升 | 设备可用性、性能率和质量率改善 | 提升5%至15% |
| 停产损失降低 | 故障响应从小时级缩短到秒级 | 降低约30% |
| 人工成本降低 | 自动重排和可解释日志减少人工分析 | 排产成本降低约70% |
| 交付风险降低 | 紧急订单和截止压力自动识别 | 减少订单延误 |

## 7.3 商业化路径
短期可采用SaaS订阅模式，面向中小型制造企业提供轻量化智能排产服务；中期可采用定制集成模式，面向大型磷化工、矿山和装备制造企业接入MES、ERP和IoT系统；长期可沉淀行业调度知识库，形成面向贵州及西南地区工业场景的数字孪生调度平台。项目的开源属性也有助于吸引高校、企业和开发者共同完善工业AI生态。

# 第八章 部署指南
## 8.1 本地开发环境
本地开发推荐使用Python 3.10以上版本和Node.js环境。后端依赖FastAPI、OR-Tools、Pydantic、SQLite、httpx和python-dotenv等库；前端依赖React、TypeScript、Vite、Tailwind、ECharts、ECharts GL和DataV组件。项目根目录应包含pyproject.toml、backend、frontend、swarm、datasets、docs和scripts等目录。
后端启动命令为：C:\Anaconda\python.exe -m uvicorn backend.main:app --host 127.0.0.1 --port 8010 --reload。前端启动命令为：进入frontend目录后执行npm install，再执行npm run dev -- --port 5174。浏览器访问http://127.0.0.1:5174即可进入系统大屏。若端口被占用，可修改端口并同步调整前端API地址。

## 8.2 环境变量配置
系统通过.env文件读取外部服务配置。豆包大模型需要配置LLM_PROVIDER=doubao、DOUBAO_API_KEY和DOUBAO_MODEL。华为云服务需要配置区域、AK/SK、项目ID、模型服务地址和数据库连接信息等。所有真实密钥只能保存在本地.env中，不得提交到Git仓库。.env.example只保留变量名和空值，供部署人员参考。

| 配置项 | 说明 | 是否必需 |
| --- | --- | --- |
| LLM_PROVIDER | llm provider名称，如mock、doubao、pangu | 可选 |
| DOUBAO_API_KEY | 豆包API Key | 真实调用必需 |
| DOUBAO_MODEL | 豆包模型名称 | 可选 |
| HUAWEI_REGION | 华为云区域 | 真实华为服务必需 |
| GAUSSDB_DSN | GaussDB连接串 | 真实数据库必需 |
| SQLITE_PATH | 本地SQLite路径 | 本地运行可选 |

## 8.3 测试与构建
项目应通过后端测试、脚本验证、前端构建和Docker构建检查。核心命令包括C:\Anaconda\python.exe -m pytest -q、C:\Anaconda\python.exe scripts/run_local_demo.py、C:\Anaconda\python.exe scripts/run_a2c_experiment.py、cd frontend; npm run build以及docker compose up --build。对于Docker或外部云服务不可用的情况，应记录真实错误，不应伪造测试结果。

## 8.4 部署注意事项
真实生产部署需要关注三类问题。第一是数据安全，工厂设备数据、订单数据和模型密钥必须隔离管理。第二是模型可靠性，大模型输出只能作为可审计建议，不应绕过CP-SAT硬约束求解器。第三是运维可观测性，系统应持续记录事件历史、Agent日志、拓扑选择、排程结果和服务连接状态，以便故障追踪和效果评估。

# 第九章 结论与展望
## 9.1 项目结论
DynaTwin-Swarm完成了从GPTSwarm通用多智能体框架到工业数字孪生智能排产系统的工程化扩展。系统围绕工业调度中的动态扰动问题，构建了状态感知、Agent协作、拓扑选择、约束求解和前端可视化的完整闭环。通过五类拓扑、11个智能体、ReflAct可审计推理、CP-SAT求解和A2C优化，系统实现了从“监控异常”到“解释异常并重排生产计划”的能力跃迁。
在工程实践中，系统坚持本地mock可运行、外部服务可替换、密钥不入库、结果不伪造的原则，既能满足竞赛演示，也能为后续真实工厂试点提供基础。当前系统已经具备豆包大模型接入、华为昇腾生态适配接口、标准数据集验证和工业大屏展示能力。

## 9.2 后续展望
后续工作将从三个方向推进。第一，接入真实工厂IoT数据流、MES和ERP系统，使FactoryState由模拟数据逐步过渡到真实生产状态。第二，面向磷化工、非煤矿山和智能制造产线建立更细粒度的工艺约束模型，包括批次约束、安全间隔、运输约束、能耗约束和人员班次约束。第三，继续优化A2C拓扑选择策略和大规模求解启发式，提高SWV20、DMU80等大规模数据集上的解质量。
从产业化角度看，DynaTwin-Swarm有潜力发展为面向西南工业企业的数字孪生调度平台。系统可以先以轻量化SaaS和本地部署形式进入中小企业，再通过定制集成服务进入大型工业企业，最终沉淀行业调度知识库和智能排产模型生态。
